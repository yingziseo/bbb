# YIYUAN 英文 TOB 博客生产、配图与自动发布工作流

最后更新：2026-07-10

## 1. 文档定位

本文档是 YIYUAN NEW MATERIALS 英文 TOB 博客的唯一现行 SOP，适用于保鲜膜和一次性餐盒两条获客内容线。

完整顺序只有一套：

```text
选题规划 → 文章生成 → 质量评审 → 二次优化 → 图片规划 → PNG 生图
→ WebP 转换 → 数据库备份 → 草稿入库与排期 → 自动发布 → 页面验证
```

硬规则：

- 不跳过评审、优化、图片规划或数据库备份。
- 不根据对话记忆判断进度，以本地文件、图片映射、SQLite、接口和页面结果为准。
- 工作流的重要规范、批次规划、状态、映射、可复用脚本和目录 README 需要提交 Git；生成的文章 HTML、PNG、工作流 WebP 和拒绝样本保持忽略。正式站点图片必须进入 `public/images/blog/`，文章必须写入 SQLite。
- 批量任务可以并发写作、评审或生图，但每个文件和文章必须有唯一归属，不能由多个代理重复处理。
- 主对话只汇总计数、缺口和验证结果；大量图片生成优先交给子代理，避免主对话积累图片上下文。

发生冲突时按以下优先级执行：

1. 用户在当前任务中的最新明确要求。
2. 本文档。
3. 当前批次规划，例如 `两个月文章规划-37-96.md`。
4. 当前批次机器可读数据，例如 `图片prompt-37-96.json`。
5. 历史规划、旧批次图片清单和旧执行记录。

`历史批次/首月30篇/` 中的文件属于早期 30 篇测试批次记录，不作为新任务模板。这里保留当时的文章规划、执行记录、图片清单和第二轮质量评审，仅用于审计。

历史记录可以保留用于审计，但不能覆盖本文档的现行数量、命名、图片和排期规则。

## 3. 新任务与继续任务隔离

### 3.1 先判断任务类型

每次开始前必须明确属于哪一种：

- **新任务**：新的文章范围、日期批次、产品线、专题或用户明确要求另开一批。
- **继续任务**：继续此前已经建立规划、文章、评审、图片映射或入库记录的同一批任务。

判断依据按顺序为：用户说明、批次状态文件、规划编号与日期范围、现有文章和图片映射。不能仅凭当前数据库文章数判断。

### 3.2 新任务必须独立建档

从下一批新任务开始，统一建立独立目录：

```text
工作流/批次/YYYYMMDD-业务主题-编号范围/
├── README.md                 # 任务目标、边界、批次状态和文件索引
├── 文章规划.md
├── 内容质量评审.md
├── 图片prompt.json           # 文章与图片的机器可读唯一映射
├── 图片prompt.md
├── 状态.json                 # 各阶段完成数、缺口、数据库和排期状态
└── scripts/                  # 仅适用于该批次的生成、校验和导入脚本
```

同时创建对应文章目录：

```text
工作流/文章/任务名称-编号范围/N.html
```

新任务要求：

- 目录名包含日期、主题和编号范围，不能使用含糊的 `新任务`、`最新` 或 `最终版`。
- README 明确任务目标、文章编号、slug 范围、计划日期、产品线数量、预计图片数和允许引用资料。
- README 和状态 JSON 都必须记录该批次的 `articleDir`，文章不能直接散放在 `工作流/文章/` 根目录。
- 状态 JSON 至少记录 `batchId`、`objective`、`articleRange`、`scheduleRange`、`expectedArticles`、`expectedImages`、各阶段状态和最后更新时间。
- 规划、评审、图片映射和批次脚本只服务该批次，不与历史批次共用可变数据文件。
- 通用规则仍引用本文档，不在每个批次复制一份容易漂移的 SOP。

### 3.3 继续任务必须复用原批次

继续此前任务时：

- 先读取原批次 README、状态 JSON、规划和图片映射。
- 复用原编号、slug、日期、提示词和输出目录。
- 通过期望清单与实际文件、数据库做差集，只完成缺失阶段。
- 不创建第二套规划、第二套图片映射或名称相近的“新版本”文件。
- 规则确需调整时，修改原批次文件并更新状态和变更说明。
- 新对话只改变会话上下文，不改变批次身份。

### 3.4 归档

批次完全发布并验证后，将批次状态标记为 `complete`。历史资料保留在其独立目录或 `历史批次/` 下，不留在现行工作流根目录与新任务混放。

## 4. 业务目标与内容边界

### 4.1 两条内容线

- 保鲜膜：PE/PVC food cling film、commercial rolls、cutter boxes、jumbo rolls、规格验收、质量控制、装箱、渠道和采购流程。
- 一次性餐盒：PP/PET/纸浆等容器、base/lid 匹配、菜单适配、运输、仓储、质检、定制和采购流程。

目标读者：

- 海外进口商和批发商。
- 包装分销商。
- 餐饮连锁、中央厨房、超市和外卖供应链采购人员。
- OEM、私标和定制包装买家。

内容目标：每篇帮助买家完成一个具体的比较、测试、计算、沟通、批准或风险判断，并自然引导到核心产品页、资料页或联系页。

### 4.2 允许引用的本站产品事实

当前只有以下两个客户本人上传、编写并确认的核心产品页可作为产品事实依据：

- 保鲜膜：`/products/commercial-cling-film-roll`
- 一次性餐盒：`/products/custom-disposable-food-containers`

其他历史 AI 产品页不能作为新文章的产品参数、应用、合规或 E-E-A-T 依据，也不主动内链。

### 4.3 公开文件的适用边界

- 产品资料：`/downloads/yiyuan-food-container-cling-film-product-catalog.pdf`
  - 可用于说明产品范围、规格、RFQ、样品和包装。
  - 不是第三方认证，不能据此写“已认证”或“全部合规”。
- 保鲜膜检验报告：`/downloads/yiyuan-cling-film-test-report-ccf-000071.pdf`
  - 仅限报告编号 `202502010694` 中明确的 PE 食品用塑料自粘保鲜膜送检样品，规格 `400 mm × 0.013 mm`。
  - 不能外推到餐盒、PVC、其他 PE 配方或规格、其他批次和所有目的市场。
- `/documents` 中 `On request` 资料是申请入口，不代表证书已经公开、签发或适用于全部产品。

### 4.4 高风险表述

没有对应证据和适用范围时禁止使用：

- `FDA certified`
- `EU approved`
- `fully compliant`
- `guaranteed food safe`
- `100% leak-proof`
- `microwave safe`
- `compostable in all markets`
- `best quality`、`cheapest factory`、`zero risk`

涉及法规、食品接触、材料性能、环保、认证或具体测试结论时：

- 优先核对当前官方一手来源。
- 说明材料、SKU、样品、批次、用途、目的市场和文件版本边界。
- 只写来源实际支持的结论。
- 餐盒文章不能用保鲜膜报告证明餐盒质量或合规。

## 5. 阶段一：选题规划

### 5.1 选题前检查

- 查询数据库和本地规划，避免重复已发布、已排期或已完成文章的主要搜索意图。
- 检查首页、产品页、分类页和既有博客 TDK，避免抢同一个主关键词。
- 确认核心产品页排序和内容没有变化。
- 确认公开 PDF 和 `On request` 文件状态。
- 优先高采购意图主题：规格、质检、RFQ、MOQ、样品、包装、装箱、成本、变更控制、索赔证据和文件审核。
- 不为凑数量写纯百科、泛趋势或空泛环保口号。

### 5.2 每篇规划字段

```text
number
publishDate
business
articleType
title
slug
primaryKeyword
longTailKeywords
searchTask
targetBuyer
allowedInternalLinks
sourcePlan
evidenceRisk
bodyImageCount
coverTheme
bodyImageThemes
conversionAction
```

批量规划必须校验：

- 编号、日期、slug 唯一。
- 两条业务线数量符合用户要求。
- 日期连续且每天一篇。
- 不从较早文章链接到尚未发布的后续文章。
- 正文图片数量为 1–3 张，并与文章复杂度匹配。

## 6. 阶段二：文章生成

### 6.1 数据字段

每篇必须生成：

```text
slug
title
excerpt
content_html
seo_title
seo_description
seo_keywords
canonical
cover_image
status
published_at
scheduled_publish_at
```

文章先保存到 `工作流/文章/任务名称-编号范围/N.html`。其中：

- 文件名使用稳定的文章编号 `N.html`。
- H1、封面、发布日期、面包屑、文章目录和底部系统 CTA 由页面层渲染，不写进正文 HTML。
- `content_html` 不得包含 H1 或手写目录。
- 图片阶段前只保留结构化 Image plan 注释；入库前替换为真实 `<figure>`。

### 6.2 正文最低要求

- 开篇直接回答搜索问题。
- 清楚说明目标买家和决策任务。
- 至少 3 个有明确任务的 H2。
- 至少一组可执行信息：表格、清单、测试步骤、计算方法、决策树或证据字段。
- 至少一个风险、限制或适用边界。
- 2–4 个自然内链；只链接允许使用的产品页、资料页、联系页和更早文章。
- FAQ 只在存在独立搜索问题时使用，可以没有 FAQ。
- CTA 必须与当前采购任务直接相关。

### 6.3 结构按文章类型变化

- 对比型：必须有对比表格和条件判断，不写绝对优劣。
- 规格型：解释尺寸、容量、厚度、宽度、卷长、卷芯、盖型或装箱，并说明选错后果。
- 定制型：覆盖 artwork、印刷、MOQ、打样、版本确认和大货一致性。
- 样品或质检型：列出检查步骤、记录方式、接受边界和升级动作。
- 成本型：区分单价、净产品量、外箱、托盘、装柜、损耗和 landed cost。
- 文件或合规型：列出产品—用途—市场—样品—报告的匹配关系，不泛化证据。

### 6.4 页面组件

按需要使用已有 class，不强制每篇套同一模板：

```text
yy-panel
yy-panel--summary
yy-panel--source
yy-note
yy-grid
yy-card
yy-table-wrap
yy-checklist
yy-figure
yy-faq
yy-cta
```

自动目录由 `ArticleToc.client.vue` 从 H2/H3 生成；正文不写目录 HTML、目录脚本或重复锚点。

## 7. 阶段三：质量评审

文章生成后先评审，不能直接生图或入库。

评审维度：

1. 搜索意图匹配。
2. 海外 B2B 买家价值。
3. SEO Title、Description 和长尾覆盖。
4. 结构和可执行信息。
5. E-E-A-T 与来源边界。
6. 产品、材料和适用范围准确性。
7. 内链和转化自然度。
8. 采购痛点与代表性场景。
9. 去 AI 味和信息密度。
10. HTML 与数据库可用性。

评审必须指出文章中的具体证据或具体缺口，不能给整批文章复制相同分数和评语。低于 8 分的文章不能进入图片阶段。

## 8. 阶段四：二次优化

每篇至少优化一轮：

- 强化一个具体采购场景。
- 强化一个买家痛点和后果。
- 增加或改善一个可执行检查、测试、计算或记录方法。
- 判断是否真的需要来源面板；需要时写具体，不需要时删除模板化面板。
- 复查食品接触、温度、微波、环保、认证和报告表述。
- 删除空泛开场、重复定义、无信息总结和统一模板句。
- 检查表格、清单和 FAQ 是否真正帮助决策。
- 检查 CTA、excerpt 和 SEO description 是否与文章任务一致。

优化后运行批量校验：

- 文件、编号和 slug 对应。
- HTML 标签闭合。
- 无 H1、无手写目录。
- H2/H3 层级可生成目录。
- 内链有效且无错误的历史 AI 产品页。
- 高风险结论有来源和适用边界。
- 图片规划数量与批次规划一致。

## 9. 阶段五：图片规划与生图

### 9.1 数量

每篇固定：

- 1 张封面图。
- 1–3 张正文图，数量由文章复杂度决定并写入批次规划。

不再使用“所有文章固定 2 张正文图”的旧规则。

### 9.2 唯一命名规则

PNG 原图：

```text
工作流/封面图/slug-cover.png
工作流/封面图/slug-01.png
工作流/封面图/slug-02.png
工作流/封面图/slug-03.png
```

WebP 成品：

```text
工作流/封面图/已经转webp/slug-cover.webp
工作流/封面图/已经转webp/slug-01.webp
工作流/封面图/已经转webp/slug-02.webp
工作流/封面图/已经转webp/slug-03.webp
```

网站正式路径：

```text
public/images/blog/slug-cover.webp
public/images/blog/slug-01.webp
public/images/blog/slug-02.webp
public/images/blog/slug-03.webp
```

不再使用 `N_0.png`、`N_1.png` 等旧数字图片名。

### 9.3 文章与图片映射

每批必须生成机器可读 JSON。每张图片至少包含：

```text
number
business
slug
kind        # cover 或 body
index       # cover、01、02、03
filename
theme
prompt
```

系统生成的 `exec-*.png` 只是临时文件，不能作为最终映射。每张合格图生成后立即按 JSON 中的 `filename` 复制到 `工作流/封面图/`。新对话恢复任务时，通过 JSON 的 `filename` 与目录做差集，不根据聊天记录猜测。

### 9.4 并发规则

- 大批量生图优先分给多个子代理并发执行。
- 每个子代理领取不重叠的编号或精确文件清单。
- 子代理只生成自己的图片，不改文章、数据库、排期或其他代理的文件。
- 主线只接收完成数量、文件清单和失败项，不回传逐张图片上下文。
- 只生成差集中缺失的图片；已有合格文件不重复生成。

### 9.5 风格与基础验收

- 16:9 横图，真实 B2B 食品包装摄影风格。
- 主题与文章对应，保鲜膜和餐盒产品形态基本合理。
- 允许出现人物、正面人物、手部操作和正常工作场景；人物或人脸本身不构成拒绝理由。
- 避免明显可读的乱码、无关品牌、公司 logo、水印、假认证、假报告章和虚构测试结果。
- 表单、报告、标签和屏幕最好空白、模糊或小到不可读。
- 不做像素级挑剔；轻微构图或材质瑕疵不反复重生成，只拒绝明显不可用图片。

### 9.6 基础检查

生图完成后只做必要检查：

- JSON 期望文件数与实际 PNG 数一致。
- 每个编号有 1 张封面和规划数量的正文图。
- 文件名、格式和尺寸可读。
- 无零字节或损坏文件。
- 抽查主题和明显禁用元素。

拒绝样本放入 `工作流/封面图/拒绝样本/`，不能占用正式文件名。

## 10. WebP 转换与正文插图

### 10.1 转换

- PNG 原图保留在工作流目录。
- WebP 统一为 `1200 × 675`，质量 `90`。
- 输出到 `工作流/封面图/已经转webp/`。
- 转换后校验数量、尺寸和文件可解码。
- 复制到 `public/images/blog/` 后，页面和数据库只引用 `/images/blog/...webp`。

### 10.2 正文替换

按每篇的正文图片数量，将 Image plan 注释替换为：

```html
<figure class="yy-figure">
  <img src="/images/blog/slug-01.webp" alt="Accurate English description of the actual scene" width="1200" height="675" loading="lazy" decoding="async">
  <figcaption>Short caption tied to the buyer task.</figcaption>
</figure>
```

要求：

- 图片顺序与 `bodyImageThemes` 和文章插入点一致。
- alt 描述真实画面，不堆关键词。
- 每张正文图包含 `width`、`height`、`loading="lazy"` 和 `decoding="async"`。
- 封面由 `cover_image` 字段渲染，不重复插入正文。

## 11. 数据库入库与排期

### 11.1 入库前

1. 检查 Git 状态并区分本任务与用户已有改动。
2. 备份 `data/yiyuan.db` 到 `data/backups/`，文件名包含任务和时间。
3. 确认 文章 HTML、PNG、WebP 和 public 图片数量完整。
4. 确认所有 slug 不冲突。
5. 确认正文已引用 public WebP 路径。
6. 确认 SEO 字段完整。

涉及数据库结构变化时，除备份外还必须提供迁移逻辑；普通文章导入不得改表结构。

### 11.2 posts 写入

批量预填充文章默认写为排期草稿：

```text
status = draft
published_at = ''
scheduled_publish_at = 中国时间计划日期的白天随机时间对应 UTC ISO
published_by_scheduler_at = ''
cover_image = /images/blog/slug-cover.webp
```

同时写入 title、excerpt、content_html、SEO 字段、canonical、created_at 和 updated_at。

### 11.3 seo_entries 同步

每篇同步：


### 11.4 排期

- 接续数据库最后一个排期日期。
- 北京时间每天一篇，默认在 `09:00–17:30` 内随机。
- 两条业务线按批次规划交替或按用户确认顺序发布。
- 排期草稿不会出现在公开 API、博客页面或 sitemap。
- 定时发布模块每 5 分钟检查到期草稿。
- 页面：`/like/post-scheduler`。
