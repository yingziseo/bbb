# AGENTS.md

## 项目协作原则

- 默认使用中文沟通，说明要直接、具体、可核对。
- 修改代码、新增模块、调整数据库、后台管理、SEO、导航、表单、页面结构或重要样式前，必须先阅读相关代码。
- 阅读代码后先汇报理解和修改方案，列清楚会改哪些模块、会影响哪些页面、是否涉及数据库/API/后台。
- 在需求细节不完整或存在歧义时，先补充问题并与用户核对，不要直接按猜测大改。
- 用户确认方案或明确说“可以”“开始”“确认”后，再执行代码修改。
- 小范围明确修复可以直接处理，但仍要先快速查看相关代码，避免盲改。

## 开发流程

1. 阅读相关代码和当前 git 状态。
2. 汇报现状、问题点和计划。
3. 与用户核对需求细节，确认边界和预期效果。
4. 用户确认后再实施。
5. 修改完成后执行必要验证，例如 `pnpm build`、接口检查、页面 200 检查或数据库结构检查。
6. 如果改动影响线上运行，构建通过后重启 `3000` 服务并验证公网访问。
7. 更新 `docs/DEVELOPMENT_LOG.md`，记录背景、改动、涉及文件、验证结果和提交状态。
8. 完成后默认提交并 `git push`，除非用户明确要求不要推送。

## Git 规则

- 提交前必须检查 `git status --short --branch`。
- 不要提交无关文件、临时文件、用户未要求提交的未跟踪文件。
- 如果工作区存在与本次任务无关的改动，要明确区分，不要误提交。
- 推送前确认 commit 内容和当前分支。
- 推送后确认 `main...origin/main` 同步状态。

## 代码修改习惯

- 优先沿用项目已有结构、组件风格、数据模型和命名方式。
- 不做无关重构，不随意移动目录，不引入没必要的新依赖。
- 涉及前台、后台、API、数据库字段时，要保持整条链路同步更新。
- 涉及数据库结构变化时，必须提供迁移逻辑，并在操作前备份线上 SQLite 数据库。
- 涉及移动端和桌面端时，要分别考虑自适应效果。
- 涉及 UI 图标、按钮、导航、弹窗、表单时，要考虑交互状态、悬浮状态、点击反馈和可关闭/可退出路径。

## 验证要求

- 代码改动后尽量执行 `pnpm build`。
- 页面改动后检查公网 URL 是否返回 200。
- API 或数据库改动后检查接口响应和数据库结构。
- 如果不能执行某项验证，需要在回复中说明原因。

## 回复要求

- 最终回复简洁说明改了什么、验证了什么、是否已 push。
- 如果有遗留问题或未提交文件，要明确指出。
